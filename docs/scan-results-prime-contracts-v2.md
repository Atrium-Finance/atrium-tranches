### High

#### 1. PrimeCDO.requestWithdraw() — stale `baseAmount` across in-call loss waterfall lets a user over-extract from the strategy at the expense of Senior/Mezz

##### Description
`TrancheVault.requestWithdraw` computes `baseAmount = convertToAssets(shares)` (or `totalAssets()` on full-drain) BEFORE calling `i_cdo.requestWithdraw(...)`. Inside `PrimeCDO.requestWithdraw`, the very first action is `_updateAccounting()`, which may apply a loss waterfall and drop the tranche TVL well below the just-quoted `baseAmount`. The flow then proceeds with the original `baseAmount` — which is now stale — to compute fee, withdraw from the strategy, and decrement Accounting via `_subFromTranche`. `_subFromTranche` silently clamps the tranche TVL at zero instead of reverting, so the size mismatch is absorbed without surfacing.

Exact code references: `contracts/core/TrancheVault.sol` lines 213–237 (`requestWithdraw` quoting `baseAmount`); `contracts/core/PrimeCDO.sol` lines 210–246 (`requestWithdraw` → `_updateAccounting` → `recordFee` → `_withdrawInstant` → `i_strategy.withdraw(netAmount)` + `recordWithdraw(netAmount)`); `contracts/core/Accounting.sol` lines 164–168 (`recordFee`), 475–479 (`_subFromTranche` clamps to 0).

Step-by-step exploit:
1. Junior tranche has TVL = 1,000 USDai-equivalent and `totalSupply` = 1,000 Junior shares; user holds all 1,000 shares.
2. `convertToAssets(1000) == 1000`. User submits `TrancheVault.requestWithdraw(1000, beneficiary)`. `baseAmount = totalAssets() = 1000` is captured.
3. `PrimeCDO.requestWithdraw(JUNIOR, 1000, ...)` runs `_updateAccounting()`, which reads `strategy.totalAssets()`. A concurrent sUSDai rate move (or any prior accumulated drift) registers as a loss → loss waterfall reduces `s_juniorBaseTVL` to e.g. 50.
4. With `feeBps = 1000` (max 10%), `feeAmount = 1000 * 1000 / 10_000 = 100`, `netAmount = 900`. `recordFee(JUNIOR, 100)` clamps Junior to 0, but `s_reserveTVL += 100` (unclamped).
5. `_withdrawInstant` calls `i_strategy.withdraw(900, sUSDai, beneficiary)` — the strategy hands the user 900 worth of sUSDai. `recordWithdraw(JUNIOR, 900)` clamps Junior (already 0) again.
6. Net effect: user took out 900 sUSDai, but the *honest* claim of their Junior position after the loss was only 50. The 850 difference comes from the strategy's pool, which on the next `_updateAccounting` shows up as a phantom loss → loss waterfall hits Mezz, then Senior. A further 100 of Senior/Mezz value silently accrues into `s_reserveTVL` and is later drained by `claimReserve`.

Impact: a user who submits a redeem just before/across an in-call loss waterfall extracts more from the strategy than the post-loss tranche actually owns; the deficit is absorbed by Mezz/Senior on the next accounting cycle. Magnitude per-call is bounded by the size of the loss applied within `_updateAccounting`. The clamping in `_subFromTranche` (and the unclamped `s_reserveTVL += feeAmount` in `recordFee`) hides the inconsistency rather than reverting.

This issue is classified High severity because (a) the trigger is permissionless — any redeem that straddles a loss event is exposed; (b) loss is real-asset, paid by Senior/Mezz holders; (c) no admin precondition required; (d) the silent clamp in `_subFromTranche` removes the natural revert that would otherwise protect callers.

### Medium

#### 1. PrimeCDO.depositOutputToken() / claimSharesWithdraw() — sUSDai live exchange rate consumed without TWAP or slippage protection

##### Description
Three call sites consume `IERC4626(i_outputToken).convertToAssets(amount)` (the live sUSDai exchange rate) directly to compute base value:

- `TrancheVault.depositOutputToken` line 154: `uint256 baseAmount = IERC4626(outputToken).convertToAssets(amount);`
- `PrimeCDO.deposit` line 183: `if (token == i_outputToken) baseAmount = IERC4626(i_outputToken).convertToAssets(amount);`
- `PrimeCDO.claimSharesWithdraw` lines 351–356: `_updateAccounting()` (which reads `strategy.totalAssets()` → live `convertToAssets`) → `baseAmount = sharesReturned * baseTVL / totalSupply`.

None of these paths accepts a `minBaseAmount` slippage parameter; none reads a TWAP. The protocol relies on sUSDai's `convertToAssets` being honest and stable within a single block. The pre-audit explicitly identifies sUSDai exchange-rate manipulation (donation to sUSDai vault, or triggering `serviceRedemptions`) as a vector that "directly skews the user's credited base amount."

Exact code references: `contracts/core/TrancheVault.sol` lines 148–168; `contracts/core/PrimeCDO.sol` lines 155–187, 342–365.

Step-by-step exploit (deposit-side sandwich):
1. Victim broadcasts `depositOutputToken(100_000 sUSDai, victim)` against the Junior tranche.
2. Attacker observes the pending tx and front-runs with a rate-inflating action on sUSDai (donate the underlying, or trigger pending sUSDai logic that elevates `convertToAssets`).
3. `convertToAssets(100_000)` returns an inflated `baseAmount = 110_000`. Victim is credited 110_000 base via `recordDeposit`; `previewDeposit(110_000)` mints shares against an inflated TVL.
4. Rate normalises in the next block; the now-realised tranche value is back to ~100_000 worth of sUSDai. The 10_000 dilution is absorbed proportionally by other Junior holders.
5. Attacker, who held a pre-existing Junior position, redeems near the now-fair price and profits from the dilution they engineered, net of the donation cost.

Step-by-step exploit (claim-side sandwich):
1. Attacker has a SHARES_LOCK request ready to claim.
2. Attacker (or an MEV searcher) inflates the sUSDai rate immediately before the claim transaction.
3. `claimSharesWithdraw` runs `_updateAccounting`, which reads the inflated `strategy.totalAssets()`, registers it as a gain, splits the gain across tranches; `baseTVL = i_accounting.getTrancheTVL(tranche)` is now elevated.
4. `baseAmount = sharesReturned * baseTVL / totalSupply` is inflated.
5. `i_strategy.withdraw(baseAmount, sUSDai, beneficiary)` sends the attacker more sUSDai than their fair share; the gap is paid by other tranche holders.

Impact: identifiable victim (a sandwich victim or a tranche peer) loses up to several percent of the affected deposit/claim, depending on the achievable rate distortion. Bounded by sUSDai's manipulability, which is external to this audit's scope.

This issue is classified Medium severity because the primary harm is one-time per-tx extraction up to ~0.5% of protocol TVL given typical sUSDai distortion costs, the trigger is permissionless but requires capital and timing, and recovery requires no admin action.

#### 2. PrimeCDO.claimSharesWithdraw() — default-zero `s_vaultToTranche` lookup silently routes unregistered vaults to SENIOR

##### Description
`s_vaultToTranche` is a `mapping(address => TrancheId)`. Solidity's default-zero return makes any unset address resolve to `TrancheId.SENIOR` (enum value 0). `claimSharesWithdraw` uses this lookup with no round-trip check:

```solidity
address vault = req.token;
TrancheId tranche = s_vaultToTranche[vault];   // unknown vault → SENIOR
...
uint256 baseTVL = i_accounting.getTrancheTVL(tranche);
uint256 baseAmount = totalSupply > 0 ? (sharesReturned * baseTVL) / totalSupply : 0;
i_accounting.recordWithdraw(tranche, baseAmount);
WithdrawResult memory wr = i_strategy.withdraw(baseAmount, i_outputToken, req.beneficiary);
ITrancheVaultBurn(vault).burnSharesFrom(address(this), sharesReturned);
```

There is no `require(s_tranches[tranche] == vault)` round-trip. So when `req.token` is an address that has never been registered (e.g., a legacy vault from a sibling deployment, or an attacker-controlled contract that the SharesCooldown was set to authorise), the function happily routes the operation against the live SENIOR tranche.

Exact code reference: `contracts/core/PrimeCDO.sol` lines 342–365.

Step-by-step exploit:
1. The protocol allows multiple CDOs to share a single SharesCooldown (the contract's `s_authorized` is owner-settable per address). Owner of CDO-2 (sibling market) authorises some address X to call `request` against the SharesCooldown shared with CDO-1.
2. X creates a SHARES_LOCK request with `req.token = arbitraryAddress` (not a registered tranche of CDO-1) and `req.beneficiary = attacker`.
3. After cooldown, attacker calls `CDO-1.claimSharesWithdraw(cooldownId)`.
4. CDO-1 reads `req.token = arbitraryAddress`, looks up `s_vaultToTranche[arbitraryAddress] = SENIOR` (default), reads `i_accounting.getTrancheTVL(SENIOR)` (CDO-1's live SENIOR TVL), computes `baseAmount = sharesReturned * seniorTVL / totalSupplyOfArbitraryAddress`, calls `i_strategy.withdraw(baseAmount, i_outputToken, attacker)`.
5. CDO-1's SENIOR TVL is decremented; sUSDai is sent to the attacker.

Impact: SENIOR drain when SharesCooldown is shared across markets and a sibling deployment's owner authorises an unexpected caller.

This issue is classified Medium severity because the attack requires a multi-market deployment with a misconfigured shared SharesCooldown — a privileged-setup precondition — but once that precondition holds, the trigger is permissionless and the drain is bounded by SENIOR TVL.

#### 3. PrimeCDO.registerTranche() — does not delete stale `s_vaultToTranche` entry on tranche rotation

##### Description
`registerTranche(TrancheId id, address vault)` writes `s_tranches[id] = vault` and `s_vaultToTranche[vault] = id` but does not delete the previous mapping when a tranche is rotated. After `registerTranche(JUNIOR, newJuniorVault)` is called over a previous Junior assignment, `s_vaultToTranche[oldJuniorVault]` still equals `JUNIOR`.

Exact code reference: `contracts/core/PrimeCDO.sol` lines 371–375.

Step-by-step exploit:
1. Owner upgrades the Junior vault: `registerTranche(JUNIOR, newJuniorVault)`.
2. `s_tranches[JUNIOR] = newJuniorVault`.
3. `s_vaultToTranche[newJuniorVault] = JUNIOR`.
4. `s_vaultToTranche[oldJuniorVault]` is **still JUNIOR** — never deleted.
5. A user (or attacker) holding shares in `oldJuniorVault` from before the rotation has a pre-existing SHARES_LOCK request whose `req.token = oldJuniorVault`. The cooldown elapses; anyone calls `claimSharesWithdraw(cooldownId)`.
6. CDO reads `s_vaultToTranche[oldJuniorVault] = JUNIOR`, looks up `i_accounting.getTrancheTVL(JUNIOR)` — but Junior TVL now reflects the NEW Junior tranche's TVL. `baseAmount = sharesReturned * newJuniorTVL / totalSupplyOf_oldJuniorVault`.
7. CDO debits the NEW Junior tranche, sends sUSDai to the original beneficiary, burns shares on the OLD Junior vault.

Impact: cross-tranche routing. The new Junior depositors lose value to claims executed against the old vault's stale shares.

This issue is classified Medium severity because exploitation requires (a) admin tranche rotation (a rare operational action) and (b) pre-existing live shares + cooldown requests in the deprecated vault (specific state alignment), but once these hold, the trigger is permissionless and the drain hits the new live tranche.

### Low

#### 1. SUSDaiStrategy._withdraw() — `wr.amountOut` is shares but Accounting subtracts assets, leaking dust into the strategy

##### Description
`_withdraw(amount, sUSDai, beneficiary)` computes `shares = i_sUSDai.convertToShares(amount)` (rounds down per ERC-4626 spec), transfers `shares` to the beneficiary, and returns `WithdrawResult.amountOut = shares`. PrimeCDO's `_withdrawInstant` and `_withdrawAssetsLock` then call `i_accounting.recordWithdraw(tranche, netAmount)` with `netAmount` (assets, not shares).

Because `convertToAssets(convertToShares(amount)) ≤ amount`, the strategy retains sUSDai dust that is no longer accounted for in any tranche. On the next `_updateAccounting`, `strategy.totalAssets()` includes the dust and registers it as a phantom gain — split across reserve, Senior/Mezz target, and Junior residual.

Exact code reference: `contracts/strategies/usdai/SUSDaiStrategy.sol` lines 151–168; `contracts/core/PrimeCDO.sol` lines 244–246, 270–273.

Per-withdrawal dust is sub-wei at large amounts, but accumulates on every withdrawal. A user requesting 10⁹ wei base sees ≤ 1 wei base value drift; over ~10⁹ withdrawals, ~10⁹ wei base = 1 base-unit drift.

This issue is classified Low severity because the per-tx dust is below 0.01% of TVL with no compounding amplifier beyond aggregation as a silent reserve gain; no direct theft path exists.

#### 2. TrancheVault.deposit() / mint() / depositOutputToken() — `previewDeposit` runs before `_updateAccounting`, exposing stale share-price MEV

##### Description
`TrancheVault.deposit(assets, receiver)` computes `shares = previewDeposit(assets)` BEFORE forwarding the deposit to `i_cdo.deposit(...)`, which is where `_updateAccounting()` (and therefore the gain split) actually runs. The number of shares minted is therefore based on a stale `totalAssets` that does not yet reflect the most recent gain split. A depositor entering during a window of un-credited yield receives shares at a depressed price and absorbs a slice of the soon-to-be-credited yield from existing holders.

Exact code reference: `contracts/core/TrancheVault.sol` lines 101–115 (`deposit`), 121–133 (`mint`), 148–168 (`depositOutputToken`).

Step-by-step exploit:
1. The strategy has accrued yield since the last `_updateAccounting` (no recent deposit/withdraw).
2. Attacker monitors the sUSDai contract for an imminent rate-update epoch or watches mempool for an existing user's `requestWithdraw` (which would `_updateAccounting` and crystallise the gain).
3. Attacker front-runs with a deposit. `previewDeposit` uses the OLD `totalAssets` and mints shares cheaply.
4. After the deposit, `_updateAccounting` credits the gain to the tranche; the attacker's freshly-minted shares now represent a slice of yield they didn't earn.

This issue is classified Low severity because the worst-case extraction is bounded to pending yield (not principal), the maximum extracted equals the gain split that would otherwise accrue to existing holders, and per the yield-only MEV cap, severity is capped at Low.

#### 3. SUSDaiAprPairProvider.getAprPair() — permissionless and state-mutating, defeats the prev/latest snapshot architecture

##### Description
`getAprPair()` is declared `external override` with no access control. The function shifts state (`s_prevSnapshot ← s_latestSnapshot; s_latestSnapshot ← (currentRate, block.timestamp)`). Anyone can call it, including with sufficient frequency to keep `s_prevSnapshot.timestamp ≈ s_latestSnapshot.timestamp`. With deltaT very small, `_computeStrategyApr` divides a tiny `growth` numerator by a tiny `deltaT` denominator, yielding noise that falls below the `convertToAssets` precision floor → `aprBase = 0`.

Exact code reference: `contracts/strategies/usdai/SUSDaiAprPairProvider.sol` lines 122–137.

A Junior depositor with a perverse incentive (more residual = bigger Junior cut of yield) can call `getAprPair()` once per block at ~30 k gas to keep the strategy APR signal at zero. This issue is classified Low severity because the impact is bounded to pending yield (Senior/Mezz lose only the floor-target portion of upside; principal is unaffected); per the yield-only MEV cap, severity is capped at Low.

#### 8. Accounting.s_srtTargetIndex / s_mzTargetIndex — compound precision drift on small deltaT

##### Description
The compound interest factor is computed as `interestFactor = (apySr * deltaT) / YEAR` and applied as `index = (index * (PRECISION + interestFactor)) / PRECISION`. With `apySr = 5e16` (5%) and `deltaT = 1` second, `interestFactor ≈ 1.6e9` — far below `PRECISION = 1e18`, so the cumulative truncation per call is approximately the last 9 decimal digits of the index update.

`updateTVL` runs on every deposit/withdraw via `_updateAccounting`. For a high-tx market, this can be thousands of calls per day. The integral of compound interest is systematically under-counted vs the analytic continuous-compounding answer — Senior's compound floor receives slightly less than promised; the residual flows to Junior.

Exact code reference: `contracts/core/Accounting.sol` lines 248–262.

Estimated cumulative under-payment over one year of 12-second blocks: 0.1–1% of yield (not principal). An adversary forcing frequent updates with dust deposits can amplify the drift.

This issue is classified Low severity because the impact is yield-only (not principal) and bounded; not directly exploitable for theft.
