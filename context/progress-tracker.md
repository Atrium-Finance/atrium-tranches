- Spec 14: wildcard permissions (`grantCall(address(0), sel, account)`) are revocable only via the matching `revokeCall(address(0), sel, account)` call. There is no per-selector "emergency revoke all wildcards" sweep. If a wildcard grant is compromised, every consumer contract that exposes `sel` is callable by the attacker until the matching revoke ships. Tracked per spec's Open Questions; consider a "freeze wildcard namespace" admin path if real-world ops needs it.
- Spec 14: `IAccessControlManager.sol` was relocated `contracts/interfaces/` â†’ `contracts/governance/interfaces/` per the spec's file structure. Single importer (`AccessControlled.sol`) updated. Future imports must use the new path `./interfaces/IAccessControlManager.sol` (from `governance/`) or `../governance/interfaces/IAccessControlManager.sol` (from siblings).
- Spec 14: license on the interface switched `BUSL-1.1` â†’ `MIT` per spec source. Concrete contract remains `BUSL-1.1`. Pattern: interface-as-public-API ships permissively, implementation ships business-source-licensed.
- Spec 14: `__UUPSUpgradeable_init()` from the spec body is dropped â€” does not exist in OZ 5.x (UUPSUpgradeable is implementation-only, no init step). Documented in-line in the contract. Same OZ 5.x substitution issue as the missing `ReentrancyGuardUpgradeable` documented in specs 06/07a.
- Spec 14: `AccessControlManager` uses **single-admin** model â€” `initialize(address admin)` only grants one `DEFAULT_ADMIN_ROLE`. Multi-admin setups must be bootstrapped post-deploy by the admin granting `DEFAULT_ADMIN_ROLE` to additional accounts. For mainnet, the deploying EOA is expected to renounce `DEFAULT_ADMIN_ROLE` to a multisig as the final deploy step.
- Spec 14: storage layout invariant for future UUPS upgrades â€” `AccessControlManager` has no own state variables, only inherited `AccessControlUpgradeable` (ERC-7201 namespaced storage) and `__gap[50]`. Any new fields must go AFTER `__gap` and consume gap slots. Reordering the inheritance chain would break the proxy.
- Spec 14: `_checkAccessAllowed(bytes4)` on consumer contracts (declared in spec 07a) was previously dead code â€” the `acm` field always pointed at zero because no concrete `AccessControlManager` existed. Spec 14 unblocks the path. Until spec 15 deploys an ACM proxy and consumers call `setAccessControlManager(...)`, every `onlyRole(...)` and `_checkAccessAllowed(...)` invocation still reverts (`acm.hasRole(...)` reverts on zero-address external call). Operationally: deploy ACM first, then call `setAccessControlManager(acm)` on every consumer before exercising any role-gated path.
- Spec 10'': `setCooldownDisabled` is gated by `COOLDOWN_WORKER_ROLE` (same role as `transfer`). Rationale: the Strategy that grants `transfer` access also drives cooldown duration changes via its own `setCooldowns(...)` body which calls `silo.setCooldownDisabled(...)` atomically. Separating the toggle into a dedicated `EMERGENCY_ROLE` would force admin to grant two roles to Strategy for no operational gain. Trade-off: a compromised Strategy can both lock the silo (call `transfer`) AND empty it instantly (toggle `setCooldownDisabled(true)` then `finalize`). If Strategy compromise is a credible threat model, split into an `EMERGENCY_ROLE`. Tracked per spec's Open Question.
- Spec 10'': a single `ERC20Cooldown` silo currently serves exactly one Strategy concrete (`UsdaiStrategy`). The per-token `cooldownDisabled` flag and per-token `_activeRequests` keying are forward-compat for sharing one silo across multiple Strategy concretes (one per underlying asset), but the MVP deployment is 1:1. If multiple Strategies ever need a shared silo, role-grant matrix becomes "every Strategy holds `COOLDOWN_WORKER_ROLE`" â€” they can each toggle each other's `cooldownDisabled` via the per-token map, which may be undesirable; revisit if multi-strategy becomes real.
- Spec 10'': no `__gap` declared on `ERC20Cooldown.sol`. `code-standards.md` mandates `uint256[50] __gap` on upgradeable base contracts; this is a concrete leaf inheriting from `CooldownBase` (which itself has no own state but inherits `AccessControlled`'s `__gap[48]`). The spec source omits a gap; followed verbatim. If `ERC20Cooldown` ever needs a derived subclass, add `uint256[50] __gap` then.
- Spec 10'': no cancel surface. Atomic with the Atrium-side withdraw flow â€” by the time tokens reach the silo, the Tranche has burned the shares and Accounting has decremented; returning tokens would require re-minting (out of scope for a silo). The emergency escape is `setCooldownDisabled(true)` + immediate finalise, which gets users their tokens but skips the lock; can't reverse a wrong-recipient `to` argument. If wrong-recipient mistakes become a real ops risk, a separate `revertTransfer(...)` admin path would need its own spec.
- Spec 10'': swap-pop pattern in `_finalize` is O(len) â€” bounded by `MAX_ACTIVE_REQUEST_SLOTS = 70`. Worst-case finalise call iterates 70 entries and performs 70 swap-pop SSTOREs, plus one `safeTransfer`. Acceptable today; if request volume materially increases (multi-strategy with heavy daily flow), revisit.
- Spec 10'': `setCooldownDisabled` writes `cooldownDisabled[token] = isCooldownDisabled` with no event. `IERC20Cooldown` and `ICooldown` declare no event for this toggle (spec source has none). Off-chain monitoring must watch the storage slot directly or infer from `Finalized` cadence. If indexer support becomes necessary, add a `CooldownDisabledChanged(IERC20 token, bool disabled)` event in a follow-up amendment.
- Spec 16a: `convertToShares(shares) > 0` defensive guard in `burnSharesAsFee` â€” spec calls it out as "may be removable". Triggers a quirk: `convertToShares` takes ASSETS not SHARES, so passing a share count is semantically odd. Effectively functions as a "can the converter execute without division-by-zero" probe. Remove or replace with explicit `totalSupply() > 0` check after the donation-attack guard (16b's `MIN_SHARES`) lands â€” at that point totalSupply collapse is structurally prevented and the guard becomes dead code.
- Spec 16a: `maxRedeem(owner)` calls `cdo.maxWithdraw(this, owner)` and converts via local `convertToShares`. Spec calls out: "Whether `maxRedeem(owner)` should fall back to the local ERC4626 calculation when CDO returns zero (avoids accidental lockout if a CDO read fails on a fresh deploy)". Currently if CDO returns 0 (e.g., on a fresh deploy before `config(...)`), `maxRedeem` returns 0 too â€” users see "cannot redeem" until the CDO is wired. This is intentional but fragile; consider a fallback if a CDO read fails. Track separately.
- Spec 16a: `TExitMode` is imported as a file-level enum and referenced bare (`TExitMode.SharesLock`); spec source writes `ICDO.TExitMode.SharesLock` which does NOT compile (file-level enums aren't members of an interface). Same pattern PrimeCDO already uses. If the spec author intended the enum to be a member of `ICDO`, that would require moving the enum inside the interface â€” out of scope here.
- Spec 16a: `ICDO.sharesCooldown()` declaration was added so `cdo.sharesCooldown()` compiles in `Tranche._withdraw`. The PrimeCDO public state variable gets `override` and the auto-getter satisfies it. Downstream integrators reading via `ICDO` now have a typed accessor â€” no more casting through `address(cdo)` or assuming PrimeCDO concrete type.
- Spec 16a: Standard ERC4626 entries (`withdraw(assets, receiver, owner)` / `redeem(shares, receiver, owner)`) now delegate to the token-routed flow with `token = asset()`. The pre-16a path called `super.withdraw(...)` which bypassed the mode router â€” that was a latent bug in the deposit foundation. Anyone reading the post-spec-03 state expecting standard ERC4626 semantics to apply will now see mode-routed semantics on the same selectors. Document for integrators.
- Spec 16a: `_withdraw` router's `Fee` branch passes the NET `baseAssets` to `cdo.withdraw(...)` â€” so Strategy releases exactly the net amount; the burned fee shares represent base-asset equivalents that **stay** in Strategy and are credited to the reserve via `accrueFee`. If a future audit prefers "always release gross, then transfer fee back" for accountability, the order would need to flip â€” but the current pattern is gas-cheaper and matches `SharesCooldown.finalizeWithFee`'s mechanics.
- Spec 16a: `previewRedeem` and `previewWithdraw` overrides take the public-fee path (`calculateExitMode(this, address(0))`). The silo special-case in spec 12 Â§4 doesn't fire because `owner = address(0)`. So the silo itself, when querying `previewRedeem` during finalisation, sees the discounted preview rather than its fee-free rate. Silo finalisation actually goes through the token-routed `redeem(token, shares, ..., owner = silo)` path which DOES call `calculateExitMode(this, owner = silo)` â€” that path short-circuits to `ERC4626` (zero fee) per spec 12 Â§4. So the silo never reads the discounted preview at the entry point; it sees the correct fee-free path. The discounted preview only matters for public callers (UI quotes).
- Spec 16a: `_withdraw` uses `_transfer(owner, silo, sharesGross)` for the SharesLock branch â€” moves shares, doesn't burn. The silo later redeems via `Tranche.redeem(token, shares, ..., owner = silo)`, which routes through `_withdraw` with mode = `ERC4626` (zero-fee short-circuit) and DOES burn at that point. This means shares continue accruing yield to the silo balance during the cooldown window. Confirmed acceptable per spec 11's design.
- Spec 16b: `OnExit` only fires from the ERC4626/Fee branch of `_withdraw`. The SharesLock branch emits no `OnExit` â€” the silo's own `RequestedCooldown` event is the canonical signal for that path. Asymmetric per design (spec author's call). If an indexer needs per-mode parity, an option is to emit `OnExit` above the silo branch (which would surface the silo path's existence even before the silo finalises) â€” tracked per spec's Open Question.
- Spec 16b: `MIN_SHARES` is enforced only on withdraw paths (`_onAfterWithdrawalChecks` runs after burn in `_withdraw` ERC4626/Fee branch and after burn in `burnSharesAsFee`). The first-deposit case isn't constrained â€” a depositor can mint any non-zero share amount without a floor check. The donation-attack vector flows through burns (last-holder dust); a deposit-side floor would only matter if the first depositor could choose a sub-`MIN_SHARES` amount AND then never withdraw, which doesn't materialise the attack. Tracked per spec's Open Question if a deposit-side floor proves necessary.
- Spec 16b: `Dynamic` is appended last in the `TExitMode` enum (decision #1). Inserting at the front (e.g., as `Dynamic = 0`) would shift the wire format of the existing three modes â€” any storage value or hash holding the integer encoding would silently change meaning. The append preserves backward compatibility for serialized `TRedemptionParams` payloads and any off-chain decoder.
- Spec 16b: `_validateRedemptionParams` is `internal pure`. The check is fully deterministic given the inputs (`params, exitMode, exitFee, cooldownSec`) â€” no SLOAD, no callee. `pure` is the right modifier; readers see the validation as a pure function of its arguments.
- Spec 16b: `_onAfterWithdrawalChecks` runs after the burn and **before** the CDO forward. Ordering rationale: post-burn so `totalSupply()` reflects the to-be-validated state; pre-forward so a revert leaves Strategy and SharesCooldown untouched. Both side-effect sinks are hard to roll back if reverted later.
- Spec 16b: meta-token `previewWithdraw(token, ...)` and `previewRedeem(token, ...)` are owner-unaware â€” they call `calculateExitMode(this, address(0))`. Same convention as the 16a non-meta previews. Public quotes don't carry a silo discount; a keeper bot that needs silo-aware previews can call `calculateExitMode(this, silo)` directly.
- Spec 16b: thin-wrapper 4-arg `withdraw`/`redeem` forward with `TRedemptionParams(TExitMode.Dynamic, 0, 0)`. Backwards-compatible with any caller written against spec 16a â€” they get the same effective behaviour (no validation) and the wire-format selectors are unchanged. Migration to opt-in validation is gradual: callers update their tx-building code to use the 5-arg overload with concrete `params` when they want slippage protection.
- Spec 16b: `quoteWithdraw` and `quoteRedeem` over-expose the internal helpers from spec 16a. Zero new storage cost, but adds ABI surface. Integrators can avoid duplicating fee math in their SDKs â€” they pass a fee value from any source (live `calculateExitMode` result, a hypothetical worst-case fee, a UI slider input) and get the corresponding shares/assets back.
- Spec 16b: meta-token `previewWithdraw` uses `Floor` rounding on `convertToAssets(token, tokenAmount, ...)`. Matches the deposit-side convention from spec 04 (favour protocol on conversion). The downstream `_quoteWithdrawShares` may round up shares to honour the fee; intentional â€” `previewWithdraw` returns the shares the user must burn to receive `assetsNet`, and rounding up shares keeps the user honest. Meta-token `previewRedeem` uses `Floor` on `convertToTokens(token, baseAssetsNet, Floor)` per spec Â§13 â€” diverges from the corresponding 16a entry-point rounding (which uses `Ceil` on `convertToTokens` so the user gets at least their entitlement). Explicit spec-source choice; live entry-point rounding does NOT shift.

## Architecture Decisions

- Use OpenZeppelin upgradeable contracts (`@openzeppelin/contracts-upgradeable`) for the tranche base.
- Tranches inherit `ERC4626Upgradeable` directly; tranche-specific withdrawal semantics will be layered via overrides rather than a custom accounting rewrite.
- No constructors on tranche contracts â€” initialization happens through the `initializer`-guarded `initialize(...)` function.
- No access control or tranche business logic in the base contract at this stage (deferred).
- `PrimeCDO` lives at `contracts/core/PrimeCDO.sol`. It does NOT inherit `CDOComponent` (it is the CDO itself, not a component pointing back to one). Components reference `PrimeCDO` via `CDOComponent.cdo`; `PrimeCDO` verifies the back-reference at wire time.
- `PrimeCDO` uses `OwnableUpgradeable` for admin gating in this phase. Component wiring is atomic via `config(...)` (no partial setters) and is re-callable to allow re-wiring under the same owner.
- `IAccounting` and `IStrategy` are intentionally minimal â€” only what `PrimeCDO` and `Tranche` currently call. Full surfaces are deferred to their own specs.
- Deposit token-flow pattern (spec 06): the Tranche pre-approves the Strategy unlimited via `Tranche.configure()` (driven by `PrimeCDO.config(...)`), and the Strategy pulls tokens directly from the Tranche during `cdo.deposit(...)`. CDO is orchestration-only â€” it never holds the deposited tokens and never sets its own allowances. Unlimited Trancheâ†’Strategy allowance is bounded by `Strategy.onlyCDO`, so only CDO can trigger the pull. CDO emits no event on deposit (Tranche emits `Deposit` / `OnPrimeDeposit`; Strategy will emit its own once implemented).
- `PrimeCDO` uses the external-registry access-control pattern via `AccessControlled` + `IAccessControlManager` shared with Strata. Role state is not embedded in `PrimeCDO`; `onlyRole(...)` (and `_checkAccessAllowed(bytes4)`) delegate to the external `acm` contract. Same pattern is intended for every Atrium contract that needs role-gated functions. Owner gating is now 2-step (`Ownable2StepUpgradeable`): `transferOwnership(newOwner)` proposes, the new owner must call `acceptOwnership()`.
- `PrimeCDO`'s storage-layout baseline shifted on 2026-05-21 via spec 07a (Initializable â†’ Ownable â†’ Ownable2Step â†’ ReentrancyGuard (non-upgradeable, ERC-7201) â†’ AccessControlled (acm + twoStepConfigManager + __gap[48]) â†’ PrimeCDO own (5 slots + __gap[50])). Further amended on 2026-05-21 via spec 07b: PrimeCDO own tail extended to `5 slots + actionsJr/actionsMezz/actionsSr (3 packed-bool slots) + __gap[47]`. All future upgrades start from this layout. No `PrimeCDO` proxy was live before either shift.
- Pause is gated by `PAUSER_ROLE` (delegated to the external `AccessControlManager` via `AccessControlled`) and stored per-tranche on `PrimeCDO` as `TActionState` structs (one per Junior/Mezzanine/Senior). Default DISABLED on every tranche/category, mirroring StrataCDO â€” forces a deliberate enable step post-deploy and is audit-friendly.
- Pause checks are performed only at CDO level (in `CDO.deposit(...)` and `CDO.withdraw(...)`), not at Tranche level. Accepted trade-off: a paused-tranche user call burns ~150k gas through the Tranche flow before the CDO reverts. Re-evaluate if pause becomes a frequent operational lever.
- `TrancheKind` enum lives in `IAccounting.sol` (file-level enum, shared vocabulary). Both `PrimeCDO` and `Accounting` import it. Resolves the spec-07b open question about promoting the enum to a shared location.
- `Accounting` dispatches address-keyed methods via three SLOADs against `cdo.jrVault() / mezzVault() / srVault()` â€” no duplicated `_kindOf` table on Accounting. The CDO's wiring is the single source of truth for which address is which kind.
- APR Feed integration is **pull-primary + admin push override**. `Accounting.updateAccounting(...)` (future spec 08b) pulls fresh APYs from `_feed.fetchAPYs()` whenever the feed is set; `setAPYs(base, bench)` exists for admin override of the next cycle. Setting `_feed` to zero disables pull mode (push-only thereafter).
- Reserve receives auto-allocation from `netGain` on every `updateAccounting(...)` (configurable `reserveRate`, default 0 â†’ no auto-allocation) **and** from fee accrual via `accrueFee(...)`. `reduceReserve(...)` is `onlyCDO` and supports both treasury withdrawal and per-tranche redistribution.
- `PrimeCDO.updateAccounting()` is `onlyTranche` and forwards `_strategy.totalAssets()` to `_accounting.updateAccounting(...)`. The CDO is a thin orchestrator on this path â€” no logic on top of forwarding.
- **Accounting math model (spec 08b)**: Senior target compounds via an index pattern (`seniorIndex`) so per-update Senior gain is `srtNavT0 Ã— (indexT1 / indexT0 - 1)`, decoupling Senior yield from update cadence. The risk premium uses `RP_nominal = riskX + riskY Ã— (TVL_sr / pool)^riskK` with `pool = TVL_jr + TVL_mz + TVL_sr` (three-tranche pool). Senior target APR = `max(aprTarget, aprBase Ã— (1 - RP_nominal))`. Reserve cut is taken **first** off `gain`, so the Senior target sum compares against `gain_after_reserve`. Residual = `gain_after_reserve âˆ’ srTargetGain âˆ’ mzBase âˆ’ jrBase` covers both `RP_delivered` and any excess yield; it's split via `share_jr = Î± Ã— TVL_jr / denom`, `share_mz = TVL_mz / denom`, `denom = Î± Ã— TVL_jr + TVL_mz`. Junior absorbs all dust from floor-divided shares. When `denom == 0` (no Mz, no Jr) the residual goes back to the reserve. A `navT1 == jr + mz + sr + reserve` invariant is asserted post-split.
- **Linear compounding window** for the Senior index (spec 08b): `indexT1 = indexT0 Ã— (1 + aprSrt Ã— dt / YEAR)`. Approximation acceptable because Atrium expects frequent updates; long-quiescent windows under-pay Senior by a small margin (favors protocol).
- **PRBMath dependency** (spec 08b): `@prb/math@^4.1.1` is now a runtime dependency for `UD60x18` fixed-point math (used for `aprBase`, `aprTarget`, `aprSrt`, `riskX/Y/K`). Imports are explicit per-symbol: `UD60x18` from `ValueType.sol`, `ud` from `Casting.sol`, `mul`/`pow` from `Math.sol`. The Solidity 0.8.19+ `using ... global` directives in `ValueType.sol` provide operator overloading (`+`, `-`, `<`, `>`, `>=`) and the `.unwrap()` method on the UD60x18 newtype.
- **APR Feed model (current 08b)**: pull-only via a Chainlink-style `IAPRFeed.latestRoundData()` returning a `Round { roundId, aprTarget(int64 SD7x12), aprBase(int64 SD7x12), updatedAt }`. Accounting normalises SD7x12 â†’ UD60x18 (`Ã— 10^6`) and clamps to `[0%, 200%]`. The oracle (or any UPDATER_FEED_ROLE holder) signals new rounds by calling `Accounting.onAprChanged()`, which refreshes accounting and pulls the new pair. `updateAccounting(...)` itself does NOT pull APRs â€” the hot path stays cheap; the Senior index compounds correctly using the cached `aprSrt` between pulls.
- **`Accounting` access control**: `onlyRole(UPDATER_FEED_ROLE)` for `onAprChanged`; `onlyRole(UPDATER_STRAT_CONFIG_ROLE)` for `setRiskParameters`; `onlyOwner` for `setAprPairFeed`, `setReserveRate`, `setLeverageAlpha`. CDO-driven hooks (`updateAccounting`, `updateBalanceFlow`, `accrueFee`, `reduceReserve`) remain `onlyCDO`. `Accounting` now inherits `AccessControlled` directly â€” its own owner/acm independent of the CDO's.
- **Build flag**: `viaIR: true` is enabled in `hardhat.config.ts` to keep the spec-verbatim `calculateNAVSplit` body compilable under solc 0.8.35. Optimizer runs unchanged at 200.
- **Coverage threshold is hardcoded** (`PrimeCDO.MIN_COVERAGE = 1.05e18`, spec 09). Immutable, no admin override; auditors verify once and no governance attack surface. Trade-off: protocol cannot react to new market conditions without redeploying. If tuning becomes necessary, a follow-up spec can lift the value to storage with role-gated setter.
- **Senior withdraw is unrestricted** (spec 09). Withdrawing Senior raises `pool / sr` (numerator stays, denominator shrinks) so it can never reduce coverage. Senior's per-owner share balance, enforced by the Tranche vault, is the only limit.
- **Jr/Mz share a single coverage buffer** (spec 09). Both tranches sit subordinate to Senior in the loss waterfall â€” they jointly provide Senior's protection. Splitting the buffer would require admin config and introduce a governance surface. Race-condition first-come-first-served on actual withdrawal is acceptable because the buffer protects Senior (not Jr/Mz specifically), no single user can drain it in one tx, and steady-state distribution can be modeled if Jr consistently wins.
- **Coverage gate is dual-enforced** (spec 09): the view (`maxDeposit` / `maxWithdraw`) returns 0 / a capped value when the gate is violated, AND the body (`deposit` / `withdraw`) reverts `CoverageBelowMinimum`. UI cannot trick users into burning gas on doomed transactions, and a malicious UI cannot bypass the gate either.
- **Coverage gate location is CDO, not Accounting** (spec 09). Accounting's per-tranche `maxDeposit` / `maxWithdraw` stubs remain `NotImplemented()` â€” coverage policy lives where it's enforced. Accounting stays a pure-calculation surface.
- **`coverage()` uses stale snapshot** (spec 09). Reads `_accounting.totalAssetsT0()` (last-recorded TVLs, no fresh strategy fetch). Acceptable because every state-changing entry refreshes accounting upstream via the Tranche layer; off-chain consumers can simulate `updateAccounting` if they need fresh values.
- **Atrium base asset = USDai** (spec 10). Alternative tokens accepted by Strategy must be ERC-4626 vaults whose underlying asset is USDai. The reference alternative is **sUSDai**. The supported-token list convention is `[sUSDai, USDai]` â€” alternative first, base second.
- **Strategy holds sUSDai internally to earn yield** (spec 10). Both deposit paths (USDai â†’ stake into sUSDai vault, sUSDai â†’ hold as-is) terminate with the strategy holding sUSDai (yield-bearing form). Yield accrues via sUSDai's exchange rate; the strategy reports `totalAssets()` in USDai units by combining its raw USDai balance with `convertToAssets(sUSDai balance)`.
- **Withdraw returns sUSDai instantly** (spec 10). Atrium does NOT honour USDai's 30-day async-redeem cooldown â€” the strategy hands sUSDai (yield-bearing) directly to the receiver. The user, if they want USDai, must perform the redeem themselves downstream via the underlying protocol. Atrium's own protocol-level user-side delay is `SharesCooldown` (deferred spec); there is no other cooldown on the withdraw path.
- **`Strategy.sol` is abstract; concrete `UsdaiStrategy` is a future spec** (spec 10). Three reasons: (1) the USDai integration is non-trivial (ERC-4626 deposit, exchange-rate read, allowance management) and putting it in the base contract would couple the foundation to a specific underlying; (2) future protocol targets can inherit the same `Strategy` base without rewriting access control, storage, or registry logic; (3) the skeleton lands fast, the concrete is its own spec scope.
- **Strategy supported-token registry shape** (spec 10): `IERC20[]` not `mapping`. Small ordered lists (typically 2-3 tokens) are cheaper to enumerate than a mapping, and `getSupportedTokens()` returns the array directly to off-chain consumers. Swap to a mapping + enumeration array if list grows beyond ~10 entries.
- **`baseAsset` separated from `_supportedTokens`** (spec 10). Stored in a dedicated slot to save one array lookup per conversion call. The base asset is still included inside `_supportedTokens` so off-chain consumers see the complete accepted list.
- **Three-range per-vault coverage table** (spec 11): each vault has a `TExitUpperBounds { p0, p1, r0, r1, r2 }` where `r* = (feeBps, sharesLock)`. Selection at `requestRedeem` time, snapshotted into the user's `TRequest`. Coverage and per-day fee both encoded in `uint256` 1e18 (matches `PrimeCDO.coverage()` and the rest of the codebase). `sharesLock` is `uint32` seconds. Slot pack across `(p0, p1, feePpm)` was sacrificed for precision per spec Q5 â€” `TExitUpperBounds` consumes 8 slots per vault.
- **`requestRedeem` is `COOLDOWN_WORKER_ROLE`-gated** (spec 11). `PrimeCDO` holds the role and is the only worker. Operationally: deploy `AccessControlManager`, grant `COOLDOWN_WORKER_ROLE` to `PrimeCDO`, then spec-12's `PrimeCDO.cooldownShares(...)` glue can drive the silo.
- **Cancel returns shares without fee; early exit pays proportional fee** (spec 11 Q6 / Q9). `cancel` and `finalizeWithFee` are intentionally separated: cancel is the "I changed my mind" path (zero cost), `finalizeWithFee` is the "I want my shares out faster than the lock allows" path (`feePerDay Ã— daysLeft`, ceiling). Per-day fee capped at 1% (`MAX_FEE_PER_DAY = 0.01e18`).
- **Slot caps adopted verbatim from upstream pattern** (spec 11 Q8): `MAX_ACTIVE_REQUEST_SLOTS = 70` (merge into last entry on overflow â€” keeps `finalize` / `balanceOf` loops bounded), `PUBLIC_REQUEST_SLOTS_CAP = 40` (revert when `initialFrom != to` â€” anti-spam protection for the external receiver). Both `internal constant` on `CooldownBase`.
- **Cooldown disabled sentinel** (spec 11): a vault configured with `bounds.p1 == 0 && bounds.r2.sharesLock == 0` is treated as "cooldown not configured". `_isCooldownActive(vault)` returns false; `finalize` skips the `req.unlockAt > at` check and clears any historically-locked entry immediately. This is the emergency-unlock / no-cooldown deployment path.
- **`SharesCooldown` and PrimeCDO are sibling concretes** (spec 11). PrimeCDO calls into the silo via `sharesCooldown.calculateExitParams(...)` and `sharesCooldown.requestRedeem(...)` (spec 12). The silo calls back into the tranche via `vault.redeem(token, shares, receiver, owner = address(this))` â€” when the owner is the silo, PrimeCDO's `calculateExitMode(...)` short-circuits to plain ERC4626 redeem (no fee, no cooldown). This keeps finalisation a single-tx flow.
- **`TExitMode` lives on `ICDO`** (spec 12). File-level enum `{ ERC4626, SharesLock, Fee }` adjacent to the `ICDO` interface so every caller (Tranche, indexer, integrators) shares the vocabulary. Tranche imports it from `ICDO.sol` when wiring its mode-routing logic.
- **Silo recognition is unforgeable** (spec 12). `PrimeCDO.withdraw(...)` infers "this is a silo finalisation" by comparing `owner_` against the CDO's own `sharesCooldown` storage. A `bool isSiloFinalising` flag from Tranche would be lie-coordinated â€” a misbehaving Tranche could fake it and skip cooldowns for normal users. Owner-address comparison can't be faked.
- **`MAX_EXIT_FEE = 0.1e18` is hardcoded** (spec 12). Bump-as-code only, consistent with `MIN_COVERAGE`. Round number, defensible: high enough to deter exits during a crunch, low enough that admin compromise can't convert exit fees into outright confiscation.
- **Mode routing lives on Tranche; gate enforcement lives on CDO** (spec 12). Three withdraw modes (`ERC4626` / `SharesLock` / `Fee`) have three different share-manipulation patterns (burn full / transfer-to-silo / burn-with-fee-split). Only Tranche owns shares, so mode routing belongs there. CDO sees two entrypoints (`withdraw` / `cooldownShares`) and applies pause + coverage gates uniformly across both.
- **`cooldownShares` has no coverage gate** (spec 12). Coverage already steers users into the silo's worst range at low buffer levels (long lock, high fee). A second hard gate would just block them from the throttle that exists for that exact case. Pause check is present â€” without it the silo could queue requests behind the back of an active pause and finalise them later.
- **`accrueFee` and `updateBalanceFlow` overloads are `onlyTranche`** (spec 12). Conservative â€” tightens if a non-Tranche caller proves necessary. `accrueFee` accepts a `tranche` parameter but does NOT enforce `tranche == msg.sender`; Tranche is a protocol-owned contract trusted as a unit (Open Question â€” may be tightened).
- **Atomic three-field `setExitFees(jr, mz, sr)`** (spec 12 Non-Goal). No per-tranche individual setters â€” every change writes all three. Atomic admin action; off-chain runbook always specifies the full triple. Each value capped at `MAX_EXIT_FEE`.
- **Reserve is a logical bucket inside Strategy's token pool** (spec 13). Reserve is NOT a separate vault â€” it's an abstract slice of the same token pool the tranches share. Strategy stakes, harvests, and unstakes one balance; Accounting carves that balance into Jr/Mz/Sr/Reserve buckets for share-price math. Keeps the integration with the underlying yield protocol simple.
- **`RESERVE_MANAGER_ROLE` for `reduceReserve`, not `onlyOwner`** (spec 13). Owner is governance (typically a multisig with high-friction key rotation); reserve extraction is an ops task (monthly skim, quarterly profit-take, occasional treasury rebalance). Giving ops a focused role lets them do their job without holding the multisig key.
- **Treasury is owner-mutable via `setReserveTreasury(addr)`** (spec 13). No "unchanged" guard â€” treasury rotation is rare and a no-op write is harmless. Zero is rejected to keep the `treasury == address(0)` precondition in `reduceReserve` meaningful (otherwise an admin could create a window where treasury was nulled mid-migration and a reserve manager fires a transfer to the zero address, burning the tokens).
- **`Accounting.reduceReserve` uses the 4-arg shape `(baseAssets, jrAmount, mzAmount, srAmount)`** (spec 13). Future-proofs for the 08c distribute counterpart (returning reserve to tranche NAV) â€” spec 13's CDO entry passes the three trailing args as zero; 08c will populate them.
- **Floor rounding on `convertToAssets` in `reduceReserve`** (spec 13). When converting `tokenAmount â†’ baseAssets`, floor favours protocol â€” admin can't accidentally drain more base-asset equivalent than the actual token movement justifies. The alternative (Ceil) would over-debit the bucket and could push it negative on rounding alone.
- **Single-token, single-amount per `reduceReserve` call** (spec 13 Non-Goal). No multi-token batched extraction, no sweep-all helper, no per-token rate limits. Each call drains one `(token, amount)` pair; orchestration across tokens is the caller's responsibility (script or multisig batch).
- **`distributeReserve` (returning reserve to tranche NAV) is pushed to spec 08c** (spec 13 Non-Goal). Distribute is semantically tied to the recovery flow alongside the loss waterfall, so it belongs with `_applyLossWaterfall`'s body â€” not with the treasury-skim entry point shipped in 13.
- **Both ACL patterns supported on the same registry** (spec 14). `AccessControlManager` backs both `onlyRole(bytes32)` (role-based ACL, via OZ `hasRole`) AND `_checkAccessAllowed(bytes4)` (call-based ACL, via the new `isAllowedToCall`). Single contract handles both vocabularies; consumers pick the granularity that fits each function. Atrium uses `onlyRole(...)` everywhere today; call-based is shipped now so future contracts have the fine-grained option without an ACM redeploy.
- **UUPS-upgradeable; admin-gated upgrades** (spec 14). `AccessControlManager` extends `UUPSUpgradeable`; `_authorizeUpgrade(...)` is gated by `DEFAULT_ADMIN_ROLE`. The deploying EOA can rotate to a multisig via standard OZ `grantRole(DEFAULT_ADMIN_ROLE, multisig) â†’ renounceRole(DEFAULT_ADMIN_ROLE, deployer)`. Upgrade migration story: patch-in-place via `upgradeTo(V2)` is preferred over redeploy-and-rewire (which would require `setAccessControlManager(newACM)` on every consumer).
- **Wildcard namespace via `contractAddress == address(0)` in `grantCall`** (spec 14). `_roleForGlobal(sel)` is the internal-only entry that packs `(0, sel)` into a `bytes32`; `isAllowedToCall(account, sel)` checks contract-specific first then falls back to wildcard. `_roleFor` itself reverts on zero inputs to keep `grantCall(0, 0, x)` from collapsing into the role-based namespace. Use sparingly â€” useful for "hot-wallet keeper rotation" patterns where a bot needs the same selector across many strategies.
- **Single-admin model on `initialize`** (spec 14). `initialize(address admin)` grants exactly one `DEFAULT_ADMIN_ROLE`. Multi-admin setups happen post-deploy. Simpler init surface; mainnet runbook always handover from the deploying EOA to a multisig as the final step.
- **`_roleFor` encoding: address in high 20B, sel in low 4B** (spec 14). Conventional shape for `(contract, selector)`-keyed permissions; 8 zero bytes of padding in between. Indexers can decode role identifiers without an out-of-band schema. `_roleForGlobal` reuses the same low-bits encoding, with high 20B implicitly zero.
- **Interface relocated to `contracts/governance/interfaces/`** (spec 14). `IAccessControlManager.sol` now lives next to its implementation rather than under the generic `contracts/interfaces/` folder. Sets the pattern for future governance interfaces (ACM-V2, etc.). License changed to `MIT` on the interface (concrete stays `BUSL-1.1`).
- **Zero-cooldown short-circuit on `ERC20Cooldown.transfer`** (spec 10''). When `cooldownSeconds == 0`, the silo skips slot creation entirely â€” `safeTransferFrom(worker â†’ recipient)` then `Finalized` emission for indexer parity. Default Atrium configuration starts every tranche cooldown at zero; under that config, every withdraw is a single-tx pass-through with no silo accounting. Slot creation only kicks in when an admin sets a non-zero per-tranche cooldown.
- **No cancel â€” tokens flow one-way through `ERC20Cooldown`** (spec 10''). By the time tokens reach this silo, the Atrium-side withdraw flow has already completed (shares burned in Tranche, Accounting decremented). Returning would require re-minting or crediting Accounting back â€” both out of scope for a silo. Emergency exit is `setCooldownDisabled(true) â†’ finalize` rather than a per-request cancel.
- **Per-token `cooldownDisabled` flag, `COOLDOWN_WORKER_ROLE`-toggled** (spec 10''). Keyed by token (not globally) so one silo can serve multiple Strategy concretes (one per underlying asset) and toggle each independently. The toggle role is the same `COOLDOWN_WORKER_ROLE` that grants `transfer` access â€” Strategy is the one orchestrating cooldown duration changes via its own `setCooldowns` body. Separating into an `EMERGENCY_ROLE` would gain nothing operationally; tracked as Open Question if Strategy compromise becomes a credible threat.
- **Worker-pull pattern on `ERC20Cooldown` zero-cooldown branch** (spec 10''). The silo never holds tokens in the zero-cooldown path â€” calls `safeTransferFrom(msg.sender = worker, to, amount)` to push direct from Strategy â†’ user. Requires Strategy to keep its silo allowance set at all times (established once at `UsdaiStrategy.initialize` via `forceApprove(silo, type(uint256).max)`, never revoked). Standing-allowance pattern is bounded by the silo's `onlyRole(COOLDOWN_WORKER_ROLE)` gate â€” only the Strategy can trigger the pull.
- **Slot caps reused from `CooldownBase`** (spec 10''). `MAX_ACTIVE_REQUEST_SLOTS = 70` (merge into last entry at the cap to keep `finalize` / `balanceOf` loops bounded), `PUBLIC_REQUEST_SLOTS_CAP = 40` (revert when `initialFrom != to` â€” anti-spam for external receivers). Same values as `SharesCooldown` so audit reviewers see consistency across both silo flavours.
- **`ERC20Cooldown` deployment is 1:1 with `UsdaiStrategy`** (spec 10'' Open Question). MVP target is one silo per Strategy. Per-token storage shape is forward-compat for shared silos, but the deployment / role-grant matrix is currently 1:1. Multi-strategy sharing is unsigned-off.
- **All Tranche withdraw entries converge on `_withdraw` with explicit mode payload** (spec 16a). Standard ERC4626 (`withdraw(assets, receiver, owner)` / `redeem(shares, receiver, owner)`), token-routed (`withdraw(token, assets, receiver, owner)` / `redeem(token, shares, receiver, owner)`), and silo-driven finalisation all route through the same 10-arg internal `_withdraw(token, caller, receiver, owner, baseAssets, tokenAssets, sharesGross, exitMode, exitFee, cooldownSec)`. Mode payload (`exitMode, exitFee, cooldownSec`) is computed once at the entry and threaded through â€” branch dispatch is explicit and visible at the call site. No path bypasses the mode router.
- **Standard ERC4626 entries delegate to token-routed flow with `token = asset()`** (spec 16a). Eliminates the latent bug where `super.withdraw/redeem` skipped mode routing. Single point of truth for exit semantics; integrators using either entry get identical CDO consultation.
- **SharesLock branch transfers shares (not burn); silo redeems via standard ERC4626 path on finalise** (spec 16a). Shares continue accruing yield to the silo balance during the cooldown window. When the silo finalises, it calls `Tranche.redeem(token, shares, ..., owner = silo)`, which short-circuits in `calculateExitMode` (spec 12 Â§4) to `ERC4626` mode and burns the shares at that point. Two distinct burn moments depending on path; both bookkept by the CDO via `updateBalanceFlow`.
- **Public preview/quote uses `calculateExitMode(this, address(0))`** (spec 16a). Owner-unaware by design â€” preview values are a public quote, not a per-user calculation. Silo special-case (`owner == sharesCooldown`) doesn't fire with zero owner; preview always reflects the public mode (SharesLock fee or per-tranche fallback). Quoting a silo-discounted price to the public would mislead. The silo itself never reads the discounted preview at the entry point â€” its finalisation path enters via `redeem(token, shares, ..., owner = silo)` and gets the fee-free `ERC4626` route directly.
- **`Fee` branch keeps fee assets in Strategy; reserve credit via `accrueFee`** (spec 16a). User burns `sharesGross`; Strategy releases only the NET `baseAssets`; the gross-vs-net delta stays in Strategy and is credited to the reserve bucket via `cdo.accrueFee(...)`. No tokens move during fee accrual â€” just an Accounting-side bookkeeping update. Saves one ERC20 transfer per fee'd withdraw.
- **Allowance spent up-front in `_withdraw`** (spec 16a). Before the mode branch, `_spendAllowance(owner, caller, sharesGross)` runs when `caller != owner`. For SharesLock, the silo finalises on behalf of the recipient later (silo emits the burn-and-release at its own pace). Spending the caller's allowance now means the silo can't be tricked into finalising with a stale or revoked allowance â€” the allowance contract is at the share-handoff moment, not at the asset-release moment.
- **`burnSharesAsFee` uses fee-free `super.previewRedeem`** (spec 16a). This entry IS the fee accrual; routing through the local fee-aware `previewRedeem` would double-discount. Cost: callers must understand which entry charges what â€” the fee silo (spec 11) intentionally accrues via this entry rather than through `_withdraw`'s `Fee` branch.
- **`Dynamic` appended last to `TExitMode`; CDO never returns it; only consumed as a caller sentinel** (spec 16b). Append-only enum extension preserves the integer encoding of the existing three modes. `cdo.calculateExitMode(...)` continues to return only `ERC4626 | SharesLock | Fee` â€” the sentinel exists purely so callers can opt out of mode-slippage validation via `TRedemptionParams(TExitMode.Dynamic, 0, 0)` (the default carried by the 4-arg `withdraw`/`redeem` thin wrappers).
- **`MIN_SHARES = 0.1e18`, post-burn check, exemption for `totalSupply == 0`** (spec 16b). Donation-attack mitigation: a single dust-holder remaining below 0.1 shares while reserves inflate share price would be exploitable. Clean drain to zero (last holder withdraws their full stake) is explicitly allowed â€” they take back exactly what they put in. Hardcoded constant (no setter); convert to storage + governance setter only if real ops experience proves tuning necessary.
- **`OnExit` carries full mode metadata; replaces `OnPrimeWithdraw`** (spec 16b). Indexers learn exit mode + fee + cooldown per-event without polling CDO. Fired on ERC4626 / Fee branches of `_withdraw`. SharesLock branch retains the silo's own `RequestedCooldown` â€” per-branch asymmetry is an Open Question (could move `OnExit` above the silo branch later).
- **Meta-token previews apply the public exit fee** (spec 16b). `previewWithdraw(token, ...)` / `previewRedeem(token, ...)` consult `calculateExitMode(this, address(0))` â€” owner-unaware, silo special-case doesn't fire. Same convention as 16a's non-meta previews; public quote vs per-user calculation.
- **5-arg withdraw/redeem strict overloads + 4-arg thin-wrapper defaults** (spec 16b). Three overloads per direction (ERC4626 standard / token-routed default / token-routed strict). 4-arg defaults forward to 5-arg with `Dynamic` â€” backwards-compatible with any caller written against 16a. Opt-in validation is gradual; callers update their tx-building code only when they want slippage protection.
- **`_validateRedemptionParams` is `pure`** (spec 16b). Compiler-enforced no-state-read property; the validation is fully deterministic given its arguments. Readers see the function's purity at a glance â€” the only way `RedemptionParamsMismatch` reverts is via field-level disagreement with the CDO payload threaded in by the caller.
