The Solidity C3 linearization tolerates it.
- Spec 10: NatSpec format deviation (spec `///` → impl `/** */`) carried forward from specs 08a/08b. The single-line `/** @inheritdoc IStrategy */` form was attempted first and rejected by solc 0.8.35's parser with the same "trailing-space-captured-into-contract-name" error documented in spec 08a's session note; multi-line `@inheritdoc` blocks are the only form that compiles. Eight occurrences expanded.
- Spec 10: Existing `IStrategy` callers (`PrimeCDO.deposit`, `Tranche.configure`, `PrimeCDO.updateAccounting`, `Tranche._deposit` meta-vault path) all continue to compile against the expanded interface. The new methods (`withdraw`, `reduceReserve`, `baseAsset`) are additions; no kept method's selector or signature changed.
- Spec 11: `Tranche.burnSharesAsFee(uint256, address)` is currently a `revert NotImplemented()` stub. `SharesCooldown.requestRedeem(...)` / `finalizeWithFee(...)` paths invoke it whenever `fee > 0`, so any non-zero entry fee or any `finalizeWithFee` call will revert at runtime until the Tranche fee spec lands. Operationally: configure `vaultEarlyExitFeePerDay = 0` and pass `fee = 0` on `requestRedeem` to exercise the silo end-to-end until then.

Reconcile in the next Tranche fee spec; declaration may need to migrate (e.g. into an `ITrancheFees` sub-interface or onto a separate fee silo) once that scope is clearer.
- Spec 11: `cancel(...)` uses raw `IERC20.transfer(...)` instead of `SafeERC20.safeTransfer`. Safe today because the vault is an OZ ERC20 (always reverts on failure / returns true), but the project's `code-standards.md` mandates `SafeERC20` for ALL ERC20 transfers. Spec-verbatim retained; consider switching to `SafeERC20.safeTransfer` in a follow-up amendment.
- Spec 11: `cancel(...)` has **no fee** (spec Q6). If anti-griefing in production reveals a need to charge for cancellation (e.g., users repeatedly cancelling to grief `finalize` gas loops), this becomes a follow-up tweak. Tracked as Open Question per spec's "Check When Done".
- Spec 11: `finalize` overload set is rich (four variants: `IERC20`/`ITranche` × with/without `at`). No `finalizeMany` convenience for batched multi-vault finalisation — out of scope per spec Non-Goals. If indexer / off-chain tooling demands batched finalisation later, a wrapper contract or a follow-up overload could expose it without changing the existing surface.
- Spec 11: Concrete coverage threshold values per tranche (`p0`, `p1`) and per-day fee values are **not** set anywhere yet. `setVaultExitBounds(...)` and `setVaultEarlyExitFee(...)` must be called per-tranche post-deploy via the deploy script / governance before the silo can actually gate any withdraws. Until then, `calculateExitParams(vault, coverage)` returns the zero-initialised struct (all `r0/r1/r2` zero → `_isCooldownActive` returns false → immediate finalisation).
- Spec 11: `SharesCooldown` is owner-gated for admin setters but does NOT inherit anything beyond `AccessControlled` (no separate `Pausable` or kill-switch). If the silo needs emergency-pause semantics (e.g. to stop finalisation during an incident), use `setVaultExitBounds` with a punitive `r0` configuration or migrate by deploying a new silo and re-pointing `PrimeCDO.sharesCooldown`. - Spec 11: `SharesCooldown.initialize(owner_, acm_)` lives on `CooldownBase`. To deploy: (1) deploy `SharesCooldown`, (2) call `initialize(owner, acm)`, (3) wire onto `PrimeCDO` via `PrimeCDO.setSharesCooldown(...)` (from the spec-09 amendment), (4) `AccessControlManager` grants `COOLDOWN_WORKER_ROLE` to `PrimeCDO` (so `PrimeCDO.cooldownShares(...)` — spec 12 — can drive `requestRedeem`), (5) call `setVaultExitBounds` / `setVaultEarlyExitFee` per tranche.

- Spec 12: `accrueFee(address tranche, uint256 assets)` does NOT enforce `tranche == msg.sender` — relies on Tranche being a protocol-owned contract trusted as a unit. If a future Tranche subclass acts as a router (forwarding fee accruals for sister vaults), this is fine; if not, the spec author should decide whether to tighten the gate.
- Spec 12: `MAX_EXIT_FEE = 0.1e18` is `public constant`. Same trade-off as `MIN_COVERAGE` — immutable, no admin override, no governance attack surface. Convert to storage + role-gated setter if market evidence demands tuning.
- Spec 12: `cooldownShares` reuses `SharesCooldownUnchanged()` for the "silo not configured" branch instead of a dedicated `SharesCooldownNotConfigured()`. Acceptable today (same operational meaning — "silo wiring is in a degenerate state") but a dedicated error would be clearer to off-chain integrators. Tracked per the spec's own Open Question.
- Spec 12: `updateBalanceFlow()` and the 6-arg variant are both `onlyTranche`. If a keeper or off-chain rebalancer ever needs to drive a NAV refresh without going through a Tranche flow, the gate would need to widen (likely `onlyTranche || onlyRole(KEEPER_ROLE)`).
- Spec 12: `WithdrawalCapReached(address tranche)` error is declared per spec §3 but no spec-12 control path uses it. Placeholder for future per-tranche absolute caps (likely a spec-13 concern). If no caller materialises, the error should be removed in a follow-up sweep.
- Spec 12: `withdraw` body's first parameter `tranche` is unused (Tranche is always `msg.sender` because of `onlyTranche`). Suppressed via `tranche;`. The interface keeps the parameter for off-chain symmetry with `deposit(...)` and for the future case where an authorised non-Tranche caller wants to specify the tranche explicitly.
- Spec 12: `maxWithdraw(address tranche, address owner)` shadows the `owner()` function inherited from `Ownable2StepUpgradeable`. Solidity 0.8.35 tolerates this (no warning at the function-level scope) but a renamer (`owner_`) would be cleaner. Spec-verbatim retained.
- Spec 12 runtime gap: `withdraw`, `cooldownShares`, `accrueFee`, both `updateBalanceFlow` overloads, and `_strategy.withdraw` all compile but revert at the next-layer boundary — Accounting setters revert `NotImplemented()` until 08b is re-implemented, and `Strategy.withdraw` reverts `NotImplemented()` until 10' is implemented. Intentional — spec 12 unblocks Track A wiring without taking a dependency on Track B (yield model) or Track C (concrete strategy).
- Spec 13: `setReserveTreasury(...)` is `onlyOwner`. The spec's open question is whether the ops team should have a separate `SET_TREASURY_ROLE` instead of routing through the multisig — splitting it would let ops rotate treasury without a governance vote. Trade-off: a compromised ops key could redirect reserve outflows to an attacker-controlled address. Owner-only is the conservative default; revisit if treasury rotation cadence forces a friction-reduction.
- Spec 13: no daily/weekly cap on `reduceReserve(token, amount)` — a `RESERVE_MANAGER_ROLE` holder can drain the full reserve in one tx (bounded only by `Accounting.tvlReserve`). Acceptable today because the role is meant for a multisig or scripted ops account; if the role grant ever widens to an EOA, layer a per-window cap. Tracked per the spec's Open Question.
- Spec 13: runtime gap — `_accounting.reduceReserve(baseAssets, 0, 0, 0)` and `_strategy.reduceReserve(token, amount, treasury)` both revert `NotImplemented()` today. End-to-end execution requires spec 08b/08c (Accounting body) and spec 10' (concrete Strategy body) to land. Intentional per the spec's "Out of scope" section — spec 13 wires the entry point, not the end-to-end flow.
- Spec 13: `Accounting.reduceReserve(...)` was already declared in the 4-arg shape `(totalAmount, jrDistribute, mzDistribute, srDistribute)` from spec 08a — matching the spec's preferred "future-proof for 08c distribute" choice. Spec 13's CDO entry passes the three trailing args as zero. When spec 08c (distribute counterpart) lands, it will populate `jrDistribute / mzDistribute / srDistribute` to allocate recovered reserve back to tranche NAV via a sibling `distributeReserve` flow on the CDO.
- Spec 13: floor-rounding on `convertToAssets` favours protocol. If the alternative token (sUSDai) has a 1.027 exchange rate, draining 100 sUSDai decrements the reserve by 102 base-asset units (the 0.7 stays parked in Strategy as a small reserve top-up). Ceil would over-debit and could push the bucket negative on rounding alone — explicitly avoided.

## Architecture Decisions

- Use OpenZeppelin upgradeable contracts (`@openzeppelin/contracts-upgradeable`) for the tranche base.
- Tranches inherit `ERC4626Upgradeable` directly; tranche-specific withdrawal semantics will be layered via overrides rather than a custom accounting rewrite.
- No constructors on tranche contracts — initialization happens through the `initializer`-guarded `initialize(...)` function.
- No access control or tranche business logic in the base contract at this stage (deferred).
- `PrimeCDO` lives at `contracts/core/PrimeCDO.sol`. It does NOT inherit `CDOComponent` (it is the CDO itself, not a component pointing back to one). Components reference `PrimeCDO` via `CDOComponent.cdo`; `PrimeCDO` verifies the back-reference at wire time.
- `PrimeCDO` uses `OwnableUpgradeable` for admin gating in this phase. Component wiring is atomic via `config(...)` (no partial setters) and is re-callable to allow re-wiring under the same owner.
- `IAccounting` and `IStrategy` are intentionally minimal — only what `PrimeCDO` and `Tranche` currently call. Full surfaces are deferred to their own specs.
- Deposit token-flow pattern (spec 06): the Tranche pre-approves the Strategy unlimited via `Tranche.configure()` (driven by `PrimeCDO.config(...)`), and the Strategy pulls tokens directly from the Tranche during `cdo.deposit(...)`. CDO is orchestration-only — it never holds the deposited tokens and never sets its own allowances. Unlimited Tranche→Strategy allowance is bounded by `Strategy.onlyCDO`, so only CDO can trigger the pull. CDO emits no event on deposit (Tranche emits `Deposit` / `OnPrimeDeposit`; Strategy will emit its own once implemented).
- `PrimeCDO` uses the external-registry access-control pattern via `AccessControlled` + `IAccessControlManager` shared with Strata. Role state is not embedded in `PrimeCDO`; `onlyRole(...)` (and `_checkAccessAllowed(bytes4)`) delegate to the external `acm` contract. Same pattern is intended for every Atrium contract that needs role-gated functions. Owner gating is now 2-step (`Ownable2StepUpgradeable`): `transferOwnership(newOwner)` proposes, the new owner must call `acceptOwnership()`.
- `PrimeCDO`'s storage-layout baseline shifted on 2026-05-21 via spec 07a (Initializable → Ownable → Ownable2Step → ReentrancyGuard (non-upgradeable, ERC-7201) → AccessControlled (acm + twoStepConfigManager + __gap[48]) → PrimeCDO own (5 slots + __gap[50])). Further amended on 2026-05-21 via spec 07b: PrimeCDO own tail extended to `5 slots + actionsJr/actionsMezz/actionsSr (3 packed-bool slots) + __gap[47]`. All future upgrades start from this layout. No `PrimeCDO` proxy was live before either shift.
- Pause is gated by `PAUSER_ROLE` (delegated to the external `AccessControlManager` via `AccessControlled`) and stored per-tranche on `PrimeCDO` as `TActionState` structs (one per Junior/Mezzanine/Senior). Default DISABLED on every tranche/category, mirroring StrataCDO — forces a deliberate enable step post-deploy and is audit-friendly.
- Pause checks are performed only at CDO level (in `CDO.deposit(...)` and `CDO.withdraw(...)`), not at Tranche level. Accepted trade-off: a paused-tranche user call burns ~150k gas through the Tranche flow before the CDO reverts. Re-evaluate if pause becomes a frequent operational lever.
- `TrancheKind` enum lives in `IAccounting.sol` (file-level enum, shared vocabulary). Both `PrimeCDO` and `Accounting` import it. Resolves the spec-07b open question about promoting the enum to a shared location.
- `Accounting` dispatches address-keyed methods via three SLOADs against `cdo.jrVault() / mezzVault() / srVault()` — no duplicated `_kindOf` table on Accounting. The CDO's wiring is the single source of truth for which address is which kind.
- APR Feed integration is **pull-primary + admin push override**. `Accounting.updateAccounting(...)` (future spec 08b) pulls fresh APYs from `_feed.fetchAPYs()` whenever the feed is set; `setAPYs(base, bench)` exists for admin override of the next cycle. Setting `_feed` to zero disables pull mode (push-only thereafter).
- Reserve receives auto-allocation from `netGain` on every `updateAccounting(...)` (configurable `reserveRate`, default 0 → no auto-allocation) **and** from fee accrual via `accrueFee(...)`. `reduceReserve(...)` is `onlyCDO` and supports both treasury withdrawal and per-tranche redistribution.
- `PrimeCDO.updateAccounting()` is `onlyTranche` and forwards `_strategy.totalAssets()` to `_accounting.updateAccounting(...)`. The CDO is a thin orchestrator on this path — no logic on top of forwarding.
- **Accounting math model (spec 08b)**: Senior target compounds via an index pattern (`seniorIndex`) so per-update Senior gain is `srtNavT0 × (indexT1 / indexT0 - 1)`, decoupling Senior yield from update cadence. The risk premium uses `RP_nominal = riskX + riskY × (TVL_sr / pool)^riskK` with `pool = TVL_jr + TVL_mz + TVL_sr` (three-tranche pool). Senior target APR = `max(aprTarget, aprBase × (1 - RP_nominal))`. Reserve cut is taken **first** off `gain`, so the Senior target sum compares against `gain_after_reserve`. Residual = `gain_after_reserve − srTargetGain − mzBase − jrBase` covers both `RP_delivered` and any excess yield; it's split via `share_jr = α × TVL_jr / denom`, `share_mz = TVL_mz / denom`, `denom = α × TVL_jr + TVL_mz`. Junior absorbs all dust from floor-divided shares. When `denom == 0` (no Mz, no Jr) the residual goes back to the reserve. A `navT1 == jr + mz + sr + reserve` invariant is asserted post-split.
- **Linear compounding window** for the Senior index (spec 08b): `indexT1 = indexT0 × (1 + aprSrt × dt / YEAR)`. Approximation acceptable because Atrium expects frequent updates; long-quiescent windows under-pay Senior by a small margin (favors protocol).
- **PRBMath dependency** (spec 08b): `@prb/math@^4.1.1` is now a runtime dependency for `UD60x18` fixed-point math (used for `aprBase`, `aprTarget`, `aprSrt`, `riskX/Y/K`). Imports are explicit per-symbol: `UD60x18` from `ValueType.sol`, `ud` from `Casting.sol`, `mul`/`pow` from `Math.sol`. The Solidity 0.8.19+ `using ... global` directives in `ValueType.sol` provide operator overloading (`+`, `-`, `<`, `>`, `>=`) and the `.unwrap()` method on the UD60x18 newtype.
- **APR Feed model (current 08b)**: pull-only via a Chainlink-style `IAPRFeed.latestRoundData()` returning a `Round { roundId, aprTarget(int64 SD7x12), aprBase(int64 SD7x12), updatedAt }`. Accounting normalises SD7x12 → UD60x18 (`× 10^6`) and clamps to `[0%, 200%]`. The oracle (or any UPDATER_FEED_ROLE holder) signals new rounds by calling `Accounting.onAprChanged()`, which refreshes accounting and pulls the new pair. `updateAccounting(...)` itself does NOT pull APRs — the hot path stays cheap; the Senior index compounds correctly using the cached `aprSrt` between pulls.
- **`Accounting` access control**: `onlyRole(UPDATER_FEED_ROLE)` for `onAprChanged`; `onlyRole(UPDATER_STRAT_CONFIG_ROLE)` for `setRiskParameters`; `onlyOwner` for `setAprPairFeed`, `setReserveRate`, `setLeverageAlpha`. CDO-driven hooks (`updateAccounting`, `updateBalanceFlow`, `accrueFee`, `reduceReserve`) remain `onlyCDO`. `Accounting` now inherits `AccessControlled` directly — its own owner/acm independent of the CDO's.
- **Build flag**: `viaIR: true` is enabled in `hardhat.config.ts` to keep the spec-verbatim `calculateNAVSplit` body compilable under solc 0.8.35. Optimizer runs unchanged at 200.
- **Coverage threshold is hardcoded** (`PrimeCDO.MIN_COVERAGE = 1.05e18`, spec 09). Immutable, no admin override; auditors verify once and no governance attack surface. Trade-off: protocol cannot react to new market conditions without redeploying. If tuning becomes necessary, a follow-up spec can lift the value to storage with role-gated setter.
- **Senior withdraw is unrestricted** (spec 09). Withdrawing Senior raises `pool / sr` (numerator stays, denominator shrinks) so it can never reduce coverage. Senior's per-owner share balance, enforced by the Tranche vault, is the only limit.
- **Jr/Mz share a single coverage buffer** (spec 09). Both tranches sit subordinate to Senior in the loss waterfall — they jointly provide Senior's protection. Splitting the buffer would require admin config and introduce a governance surface. Race-condition first-come-first-served on actual withdrawal is acceptable because the buffer protects Senior (not Jr/Mz specifically), no single user can drain it in one tx, and steady-state distribution can be modeled if Jr consistently wins.
- **Coverage gate is dual-enforced** (spec 09): the view (`maxDeposit` / `maxWithdraw`) returns 0 / a capped value when the gate is violated, AND the body (`deposit` / `withdraw`) reverts `CoverageBelowMinimum`. UI cannot trick users into burning gas on doomed transactions, and a malicious UI cannot bypass the gate either.
- **Coverage gate location is CDO, not Accounting** (spec 09). Accounting's per-tranche `maxDeposit` / `maxWithdraw` stubs remain `NotImplemented()` — coverage policy lives where it's enforced. Accounting stays a pure-calculation surface.
- **`coverage()` uses stale snapshot** (spec 09). Reads `_accounting.totalAssetsT0()` (last-recorded TVLs, no fresh strategy fetch). Acceptable because every state-changing entry refreshes accounting upstream via the Tranche layer; off-chain consumers can simulate `updateAccounting` if they need fresh values.
- **Atrium base asset = USDai** (spec 10). Alternative tokens accepted by Strategy must be ERC-4626 vaults whose underlying asset is USDai. The reference alternative is **sUSDai**. The supported-token list convention is `[sUSDai, USDai]` — alternative first, base second.
- **Strategy holds sUSDai internally to earn yield** (spec 10). Both deposit paths (USDai → stake into sUSDai vault, sUSDai → hold as-is) terminate with the strategy holding sUSDai (yield-bearing form). Yield accrues via sUSDai's exchange rate; the strategy reports `totalAssets()` in USDai units by combining its raw USDai balance with `convertToAssets(sUSDai balance)`.
- **Withdraw returns sUSDai instantly** (spec 10). Atrium does NOT honour USDai's 30-day async-redeem cooldown — the strategy hands sUSDai (yield-bearing) directly to the receiver. The user, if they want USDai, must perform the redeem themselves downstream via the underlying protocol. Atrium's own protocol-level user-side delay is `SharesCooldown` (deferred spec); there is no other cooldown on the withdraw path.
- **`Strategy.sol` is abstract; concrete `UsdaiStrategy` is a future spec** (spec 10). Three reasons: (1) the USDai integration is non-trivial (ERC-4626 deposit, exchange-rate read, allowance management) and putting it in the base contract would couple the foundation to a specific underlying; (2) future protocol targets can inherit the same `Strategy` base without rewriting access control, storage, or registry logic; (3) the skeleton lands fast, the concrete is its own spec scope.
- **Strategy supported-token registry shape** (spec 10): `IERC20[]` not `mapping`. Small ordered lists (typically 2-3 tokens) are cheaper to enumerate than a mapping, and `getSupportedTokens()` returns the array directly to off-chain consumers. Swap to a mapping + enumeration array if list grows beyond ~10 entries.
- **`baseAsset` separated from `_supportedTokens`** (spec 10). Stored in a dedicated slot to save one array lookup per conversion call. The base asset is still included inside `_supportedTokens` so off-chain consumers see the complete accepted list.
- **Three-range per-vault coverage table** (spec 11): each vault has a `TExitUpperBounds { p0, p1, r0, r1, r2 }` where `r* = (feeBps, sharesLock)`. Selection at `requestRedeem` time, snapshotted into the user's `TRequest`. Coverage and per-day fee both encoded in `uint256` 1e18 (matches `PrimeCDO.coverage()` and the rest of the codebase). `sharesLock` is `uint32` seconds. Slot pack across `(p0, p1, feePpm)` was sacrificed for precision per spec Q5 — `TExitUpperBounds` consumes 8 slots per vault.
- **`requestRedeem` is `COOLDOWN_WORKER_ROLE`-gated** (spec 11). `PrimeCDO` holds the role and is the only worker. Operationally: deploy `AccessControlManager`, grant `COOLDOWN_WORKER_ROLE` to `PrimeCDO`, then spec-12's `PrimeCDO.cooldownShares(...)` glue can drive the silo.
- **Cancel returns shares without fee; early exit pays proportional fee** (spec 11 Q6 / Q9). `cancel` and `finalizeWithFee` are intentionally separated: cancel is the "I changed my mind" path (zero cost), `finalizeWithFee` is the "I want my shares out faster than the lock allows" path (`feePerDay × daysLeft`, ceiling). Per-day fee capped at 1% (`MAX_FEE_PER_DAY = 0.01e18`).
- **Slot caps adopted verbatim from upstream pattern** (spec 11 Q8): `MAX_ACTIVE_REQUEST_SLOTS = 70` (merge into last entry on overflow — keeps `finalize` / `balanceOf` loops bounded), `PUBLIC_REQUEST_SLOTS_CAP = 40` (revert when `initialFrom != to` — anti-spam protection for the external receiver). Both `internal constant` on `CooldownBase`.
- **Cooldown disabled sentinel** (spec 11): a vault configured with `bounds.p1 == 0 && bounds.r2.sharesLock == 0` is treated as "cooldown not configured". `_isCooldownActive(vault)` returns false; `finalize` skips the `req.unlockAt > at` check and clears any historically-locked entry immediately. This is the emergency-unlock / no-cooldown deployment path.
- **`SharesCooldown` and PrimeCDO are sibling concretes** (spec 11). PrimeCDO calls into the silo via `sharesCooldown.calculateExitParams(...)` and `sharesCooldown.requestRedeem(...)` (spec 12). The silo calls back into the tranche via `vault.redeem(token, shares, receiver, owner = address(this))` — when the owner is the silo, PrimeCDO's `calculateExitMode(...)` short-circuits to plain ERC4626 redeem (no fee, no cooldown). This keeps finalisation a single-tx flow.
- **`TExitMode` lives on `ICDO`** (spec 12). File-level enum `{ ERC4626, SharesLock, Fee }` adjacent to the `ICDO` interface so every caller (Tranche, indexer, integrators) shares the vocabulary. Tranche imports it from `ICDO.sol` when wiring its mode-routing logic.
- **Silo recognition is unforgeable** (spec 12). `PrimeCDO.withdraw(...)` infers "this is a silo finalisation" by comparing `owner_` against the CDO's own `sharesCooldown` storage. A `bool isSiloFinalising` flag from Tranche would be lie-coordinated — a misbehaving Tranche could fake it and skip cooldowns for normal users. Owner-address comparison can't be faked.
- **`MAX_EXIT_FEE = 0.1e18` is hardcoded** (spec 12). Bump-as-code only, consistent with `MIN_COVERAGE`. Round number, defensible: high enough to deter exits during a crunch, low enough that admin compromise can't convert exit fees into outright confiscation.
- **Mode routing lives on Tranche; gate enforcement lives on CDO** (spec 12). Three withdraw modes (`ERC4626` / `SharesLock` / `Fee`) have three different share-manipulation patterns (burn full / transfer-to-silo / burn-with-fee-split). Only Tranche owns shares, so mode routing belongs there. CDO sees two entrypoints (`withdraw` / `cooldownShares`) and applies pause + coverage gates uniformly across both.
- **`cooldownShares` has no coverage gate** (spec 12). Coverage already steers users into the silo's worst range at low buffer levels (long lock, high fee). A second hard gate would just block them from the throttle that exists for that exact case. Pause check is present — without it the silo could queue requests behind the back of an active pause and finalise them later.
- **`accrueFee` and `updateBalanceFlow` overloads are `onlyTranche`** (spec 12). Conservative — tightens if a non-Tranche caller proves necessary. `accrueFee` accepts a `tranche` parameter but does NOT enforce `tranche == msg.sender`; Tranche is a protocol-owned contract trusted as a unit (Open Question — may be tightened).
- **Atomic three-field `setExitFees(jr, mz, sr)`** (spec 12 Non-Goal). No per-tranche individual setters — every change writes all three. Atomic admin action; off-chain runbook always specifies the full triple. Each value capped at `MAX_EXIT_FEE`.
- **Reserve is a logical bucket inside Strategy's token pool** (spec 13). Reserve is NOT a separate vault — it's an abstract slice of the same token pool the tranches share. Strategy stakes, harvests, and unstakes one balance; Accounting carves that balance into Jr/Mz/Sr/Reserve buckets for share-price math. Keeps the integration with the underlying yield protocol simple.
- **`RESERVE_MANAGER_ROLE` for `reduceReserve`, not `onlyOwner`** (spec 13). Owner is governance (typically a multisig with high-friction key rotation); reserve extraction is an ops task (monthly skim, quarterly profit-take, occasional treasury rebalance). Giving ops a focused role lets them do their job without holding the multisig key.
- **Treasury is owner-mutable via `setReserveTreasury(addr)`** (spec 13). No "unchanged" guard — treasury rotation is rare and a no-op write is harmless. Zero is rejected to keep the `treasury == address(0)` precondition in `reduceReserve` meaningful (otherwise an admin could create a window where treasury was nulled mid-migration and a reserve manager fires a transfer to the zero address, burning the tokens).
- **`Accounting.reduceReserve` uses the 4-arg shape `(baseAssets, jrAmount, mzAmount, srAmount)`** (spec 13). Future-proofs for the 08c distribute counterpart (returning reserve to tranche NAV) — spec 13's CDO entry passes the three trailing args as zero; 08c will populate them.
- **Floor rounding on `convertToAssets` in `reduceReserve`** (spec 13). When converting `tokenAmount → baseAssets`, floor favours protocol — admin can't accidentally drain more base-asset equivalent than the actual token movement justifies. The alternative (Ceil) would over-debit the bucket and could push it negative on rounding alone.
- **Single-token, single-amount per `reduceReserve` call** (spec 13 Non-Goal). No multi-token batched extraction, no sweep-all helper, no per-token rate limits. Each call drains one `(token, amount)` pair; orchestration across tokens is the caller's responsibility (script or multisig batch).
- **`distributeReserve` (returning reserve to tranche NAV) is pushed to spec 08c** (spec 13 Non-Goal). Distribute is semantically tied to the recovery flow alongside the loss waterfall, so it belongs with `_applyLossWaterfall`'s body — not with the treasury-skim entry point shipped in 13.
